# 健康数据加密与区块链交互说明

> 本文档用于说明当前项目中“健康数据上传、加密存储、公开/私密切换、链上存证”的实际实现方式。
> 适合项目维护、课程答辩、技术讲解和二次开发时参考。

## 1. 当前目标

本项目目前采用的是**原型系统**思路：

- 健康数据需要具备基础隐私保护能力
- 数据库中尽量不保存明文健康内容
- 区块链部分只做**哈希存证**，不把完整健康档案直接写到链上
- 保留“公开数据可浏览、私密数据需解锁查看”的交互体验

这意味着当前实现更关注：

- 可运行
- 易理解
- 成本可控
- 适合展示项目思路

而不是追求医疗级别的超高强度生产架构。

---

## 2. 健康数据现在如何存储

健康数据模型定义在：`backend/app/models.py`

对应字段主要有：

- `data_content`：文本健康数据明文字段
- `encrypted_data_content`：文本健康数据密文字段
- `pdf_data`：PDF 二进制明文字段
- `encrypted_pdf_data`：PDF 二进制密文字段
- `is_public`：是否公开
- `onchain_tx_hash`：链上存证交易哈希

### 2.1 当前真正使用的存储方式

在本次修改后，**新上传或新保存的健康数据，无论公开还是私密，都会优先写入密文字段**：

- 公开文本 -> `encrypted_data_content`
- 公开 PDF -> `encrypted_pdf_data`
- 私密文本 -> `encrypted_data_content`
- 私密 PDF -> `encrypted_pdf_data`

也就是说：

- 公开数据：**不再以明文方式写入数据库**
- 私密数据：继续以加密方式写入数据库

目前 `data_content` 和 `pdf_data` 仍然保留在模型中，主要是为了：

- 兼容历史数据
- 降低这次原型改造的破坏性

但对于后续新数据，设计目标已经变成“默认密文落库”。

## 3. 当前加密技术是怎么实现的

核心文件：`backend/app/features/blockchain/encryption.py`

### 3.1 使用的加密算法

项目使用的是：

- `cryptography.fernet.Fernet`

Fernet 可以理解为一种“带认证的对称加密封装”，特点是：

- 使用方便
- 适合文本和二进制加密
- 能检测密文是否被篡改
- 很适合原型项目快速实现“加密 + 解密 + 完整性校验”

### 3.2 密钥是怎么来的

系统并不是直接把输入字符串原样作为 Fernet 密钥，而是先做一层派生：

1. 取输入字符串（比如用户私钥，或服务端固定密钥）
2. 做 `SHA-256`
3. 得到 32 字节摘要
4. 再转成 Fernet 需要的 `base64 urlsafe` 格式

对应实现：

- `build_fernet_from_private_key(...)`

### 3.3 文本与文件加密函数

- 文本加密：`encrypt_text(...)`
- 文本解密：`decrypt_text(...)`
- 二进制加密：`encrypt_binary(...)`
- 二进制解密：`decrypt_binary(...)`

这意味着：

- 手工录入的健康记录，本质是 JSON 文本被加密
- PDF 文件，本质是二进制内容被加密

## 4. 公开数据和私密数据分别用什么密钥

这是当前实现里最重要的设计点。

### 4.1 私密健康数据

私密健康数据使用：

- **用户私钥** 作为加密源

保存私密数据时：

- 前端要求用户输入私钥
- 后端校验该私钥是否属于当前用户
- 校验通过后，使用该私钥派生出 Fernet 密钥并加密内容

这样做的好处是：

- 私密数据与用户身份绑定
- 没有私钥就无法正常解密

### 4.2 公开健康数据

公开健康数据现在使用：

- **服务端公开存储密钥** 作为加密源

当前实现方式：

- 基于 `SECRET_KEY` 生成一个固定用途的存储密钥
- 在健康数据模块中通过 `_public_storage_key()` 得到

这个密钥的用途不是“让用户输入”，而是：

- 用来保护数据库静态存储内容
- 在后端读取公开数据时自动解密

所以你可以把它理解为：

- 私密数据：用户自己掌握的密钥加密
- 公开数据：系统服务端掌握的密钥加密

这样既满足了“公开可浏览”，又避免了“公开 = 明文落库”。

## 5. 用户私钥是如何校验和保存的

相关文件：

- `backend/app/features/blockchain/encryption.py`
- `backend/app/features/auth/service.py`

### 5.1 私钥校验逻辑

系统会校验两件事：

1. 输入私钥的哈希是否等于数据库里保存的 `private_key_hash`
2. 该私钥推导出来的钱包地址是否等于用户绑定的 `wallet_address`

对应函数：

- `verify_user_private_key(...)`

### 5.2 私钥是否明文保存

用户私钥本身**不直接明文保存**。

系统使用服务端密钥派生出来的 Fernet，对私钥再做一次加密保存：

- 加密函数：`encrypt_private_key_for_storage(...)`
- 解密函数：`decrypt_private_key_from_storage(...)`

这部分属于：

- “服务端受控密钥保护用户敏感信息”

## 6. 健康数据上传后的完整流程

### 6.1 上传公开健康数据

1. 用户在前端录入文本或上传 PDF
2. 前端提交 `is_public = true`
3. 后端生成公开存储密钥 `_public_storage_key()`
4. 用该密钥对文本或 PDF 加密
5. 将密文写入数据库
6. 用户后续浏览时，由后端自动解密返回前端

### 6.2 上传私密健康数据

1. 用户录入文本或上传 PDF
2. 前端提交 `is_public = false`
3. 前端要求用户输入私钥
4. 后端校验私钥与当前用户是否匹配
5. 使用该私钥派生加密密钥
6. 将密文写入数据库
7. 用户查看私密数据时，需要再次通过私钥解锁

## 7. “公开 / 私密”切换现在怎么完成

切换入口在用户端健康数据编辑表单中。

### 7.1 公开 -> 私密

执行过程：

1. 用户编辑原记录
2. 打开“私密”开关
3. 前端要求输入私钥
4. 前端把 `is_public = false` 和 `private_key` 一起提交给后端
5. 后端用用户私钥重新加密并保存

结果：

- 数据仍然是密文落库
- 后续查看时需要私钥解锁

### 7.2 私密 -> 公开

执行过程：

1. 用户先解锁私密记录
2. 前端拿到可编辑内容
3. 用户关闭“私密”开关
4. 前端提交 `is_public = true`
5. 后端不再明文保存，而是用服务端公开存储密钥重新加密并保存

结果：

- 用户端仍然可以公开浏览
- 但数据库里不是明文

这次改造的核心价值就在这里：

> “公开”只表示访问策略更宽松，不再等于“数据库明文保存”。

## 8. 区块链部分现在是怎么运作的

核心文件：`backend/app/features/blockchain/service.py`

### 8.1 当前不是“完整健康档案上链”

当前项目没有把完整健康数据直接上链。

现在链上做的是：

- **哈希存证**

也就是：

- 把健康数据相关摘要写到智能合约里
- 用于证明“这份数据在某个时间点存在过，且内容对应某个哈希”

### 8.2 当前上链时提交了什么

保存健康数据时，后端会计算：

1. `data_hash_hex`
   - 对原始内容做 `SHA-256`
2. `encrypted_digest_source`
   - 作为另一份摘要输入源，再转成 `bytes32`
3. `data_type`
   - 标记数据类型，如 `text` 或 `pdf`

然后调用合约方法：

- `storeHealthData(...)`

### 8.3 交易是怎么发送的

上链流程是：

1. 使用用户私钥恢复账户
2. 获取当前 `nonce`
3. 构造交易
4. 用用户私钥对交易签名
5. 广播到链上
6. 等待交易回执
7. 把交易哈希保存到数据库 `onchain_tx_hash`

### 8.4 为什么这么设计

这种方案适合原型项目，因为它避免了：

- 链上直接存储大文件
- 医疗数据全部上链的高 gas 成本
- 复杂的链上隐私访问控制设计

当前思路可以概括为：

- 数据主体：数据库保存
- 隐私保护：数据库加密
- 可验证性：区块链哈希存证

## 9. 当前实现的优点与限制

### 9.1 优点

- 公开数据不再明文落库
- 私密数据具备基于用户私钥的访问控制
- 区块链成本较低，适合原型展示
- 结构清晰，便于课堂或答辩讲解

### 9.2 限制

- 历史公开明文数据不会自动批量迁移，需要后续脚本处理
- 公开数据的密钥由服务端统一掌握，因此它保护的是“静态存储安全”，不是“服务端不可见”
- 区块链部分只是存证，不是完整分布式健康档案
- 当前分析逻辑仍属于规则型原型分析，不是专业医疗诊断

## 11. 如果后续继续增强，可以怎么做

可以按下面顺序逐步升级：

1. 增加“历史公开明文数据迁移脚本”
2. 将公开数据密钥拆分为独立配置项，而不是直接复用 `SECRET_KEY`
3. 增加密钥轮换机制
4. 增加健康数据操作审计日志
5. 优化链上存证策略，例如加入版本号、数据分类标签等
6. 让 AI 模块基于“公开数据 + 用户授权私密数据”做更精细的个性化分析

---

## 12. 一句话总结

当前项目的健康数据安全模型可以总结为：

> 私密数据用用户私钥加密，公开数据用服务端密钥加密，数据库默认尽量只保存密文，区块链只负责哈希存证而不直接存原始健康档案。
